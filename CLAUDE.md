# CLAUDE.md

このファイルはClaudeコード（claude.ai/code）にこのリポジトリ内での作業指針を提供する。

---

## 記述ルール（最優先）

- **説明文・ガイドライン・メモはすべて日本語で記述する**
- **コード本体・変数名・ファイル内コメントは英語を維持する**
- コード生成時も、このルールを厳守すること

---

## プロジェクト概要

**夜語り (Yogatari)** は日本語Web小説のテキスト読み上げPWA。小説家になろう・カクヨムなどのプラットフォームからCORSプロキシ経由でコンテンツを取得し、Web Speech APIまたはGoogle Cloud TTSで読み上げる。

---

## 開発方法

ビルドシステム・バンドラー・テスト・パッケージマネージャーは一切ない。アプリ全体は `index.html` 1ファイルで完結する。

- ブラウザで `index.html` を直接開く、またはHTTPサーバー（例: `python3 -m http.server`）で配信する
- コンパイル・トランスパイル・インストール不要
- **バージョン管理**: 「バージョン管理プロトコル」に従い、`index.html` 内の定数と `/docs/versions/` を同期して管理する
- **重要**: バージョン更新はユーザーの明示的指示によってのみ実行する。通常の開発中に自動で更新してはならない

---

## アーキテクチャ

アプリ全体は `index.html`（約1,780行）に収録：

- **1〜11行目**: HTMLヘッド / メタタグ（PWA・ビューポート・テーマ）
- **12〜240行目**: 全CSS（インライン `<style>` ブロック）
- **241〜591行目**: HTMLボディ — 3タブ: 読む / 履歴 / 設定
- **592行目〜末尾**: 全JavaScript（インライン `<script>` ブロック）

その他のファイル：
- `manifest.json` — PWAマニフェスト
- `icon-192.png` / `icon-512.png` — PWAアイコン
- `docs/versions/vXXX/` — バージョン管理記録（`structure.json` + `notes.md`）
- `CLAUDE.md` — Claude Code向け作業指針

### 主要グローバル状態

| 変数 | 用途 |
|------|------|
| `segs` | 現在の章のテキストセグメント配列 |
| `idx` | 現在読み上げ中のセグメントインデックス |
| `playing` | 再生状態（Boolean） |
| `queue` | バッチ/キュー再生用の章URL配列 |
| `selVoice` | 選択中のWeb Speech API音声 |
| `_gApiKey` | Google Cloud TTS APIキー |
| `_prefetchCache` | Google TTS音声BLOBのキャッシュMap |
| `currentChapterTitle` / `currentNovelTitle` | 表示中のメタデータ |
| `nextUrl` / `prevUrl` | 章間ナビゲーション |

### 音声パイプライン

2つのTTSモード：
1. **Web Speech API** (`speechSynthesis`): デフォルト。APIキー不要。`speakIdx()` で文単位再生。
2. **Google Cloud TTS**: 高品質。APIキー必須。`fetchGoogleAudio()` で取得し `_prefetchCache` にキャッシュ、`<audio id="gAudioEl">` で再生。

サイレント音声ループ（`<audio id="silentLoopEl">`）がiOSのバックグラウンド再生でAudioContextを維持するために使われる。

### テキスト処理

- `splitText(html)` → DOMノードをプレーンテキストセグメント（段落単位）に変換
- `splitSentences(text)` → 日本語句読点（`。！？…」』`）で分割
- `domToText(node)` → インライン要素を考慮した再帰的DOM→テキスト変換

### 小説サイトパーサー

- `parseSyosetuToc(doc, url)` — 小説家になろう目次の解析
- `parseKakuyomuToc(doc, url)` — カクヨム目次の解析
- `inferTocUrl(url)` — サイト判定とTOC URL構築
- `toNovelId(url)` — URLから小説IDを抽出

### CORSプロキシシステム

`PROXIES` 配列に7つのフォールバックサービスを列挙。`tryProxy(proxy, url)` はタイムアウト付きfetchをラップ。`fetchHtml(url)` はプロキシをローテーションしながら結果を `sessionStorage` にキャッシュする。プロキシ選択履歴は `localStorage` キー `yogatari_proxy_hist` に保存。

### localStorageキー一覧

| キー | 内容 |
|------|------|
| `yogatari_bk` | しおりJSON（小説 → 章 + セグメントインデックス） |
| `yogatari_skip_filters` | スキップフィルターのON/OFF状態 |
| `yogatari_custom_words` | ユーザー定義スキップワード |
| `yogatari_gkey` | Google TTS APIキー |
| `yogatari_gvoice` | 選択中のGoogle TTS音声名 |
| `yogatari_stat_*` | 読書統計（日数・作品数・話数・秒数） |
| `yogatari_proxy_hist` | プロキシ選択履歴 |

## PWA対応状況

### 実装済みのPWA要素

```html
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-status-bar-style" content="default">
<meta name="theme-color" content="#0a0908">
<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
<link rel="manifest" href="/manifest.json">
<link rel="apple-touch-icon" href="/icon-192.png">
```

- `manifest.json` を導入済み（`display: standalone`・アイコン2種・テーマカラー `#0a0908`）
- `icon-192.png` / `icon-512.png` を用意済み（`purpose: any maskable`）
- iOS Safari でホーム画面追加した場合、全画面アプリとして動作する
- `viewport-fit=cover` でノッチ・セーフエリアに対応。CSS で `env(safe-area-inset-top)` / `env(safe-area-inset-bottom)` を使用

### 未実装の要素

| 要素 | 状況 | 備考 |
|------|------|------|
| Service Worker | 対応予定なし | 本アプリはリアルタイム取得が前提のためオフライン対応は不要 |

音声キャッシュは `_prefetchCache`（Map）と `sessionStorage` で実装されているが、Service Workerによるオフライン対応はない。

## 今後の開発の進め方

### 基本方針

- **完全単一ファイル原則 (True Single File Rule)**:
  - ロジック(JS)、スタイル(CSS)、マークアップ(HTML)はすべて `index.html` 内に完結させる
  - **サービスワーカー (sw.js) は使用禁止**
  - **アプリ構成ファイルの例外**: PWA動作に必要な `manifest.json` のみ作成・維持を許可する
  - **ドキュメント生成の例外**: 「バージョン管理プロトコル」に基づく `/docs/versions/` 以下へのファイル生成（`structure.json`, `notes.md` 等）は許可する
  - **上記以外のファイル新規作成は一切禁止する**

- **/docs/versions/ の操作制約（重要）**:
  - `/docs/versions/` 配下の操作は「バージョン管理プロトコル実行時」のみ許可する
  - 通常の機能開発・修正ではこのディレクトリを変更してはならない

- **後方互換性 (Backward Compatibility)**:
  - `localStorage` の破壊的変更は禁止
  - 構造変更時は必ず Migration を実装する
    - `DATA_VERSION` を用いてバージョン管理する
    - 起動時に差分を検出し段階的に変換する
    - 不明データ・旧バージョンのデータは削除しない

- **モバイル・ファースト**:
  - iOS / Android の Safari / Chrome を前提とする
  - `env(safe-area-inset-*)` を使用する
  - 音声再生は必ずユーザー操作（クリック等）を起点とする（自動再生禁止）

- **構造変更の制御（強制確認）**:
  以下に該当する場合は、勝手に実装を進めず必ずユーザー確認を行う：
  - データ構造の変更（localStorage含む）
  - UIレイアウトの大幅変更
  - 音声制御ロジックの変更
  - 状態管理フローの変更
  - ファイル構成の変更

### 機能追加時の注意点

- **バージョン更新**:
  - ユーザーから明示的にバージョン指定（例: vXXX）があった場合のみ、「バージョン管理プロトコル」を実行する
  - `index.html` 内の `<title>` および `APP_VER` は、プロトコル実行時にのみ更新する
  - 通常の機能追加・修正ではバージョン更新を行ってはならない
- **新しいlocalStorageキー**: 必ず `yogatari_` プレフィックスを付ける
- **スキップフィルター拡張**: `yogatari_skip_filters` と `yogatari_custom_words` の両方への影響を考慮する
- **プロキシ追加**: `PROXIES` 配列に追記するだけでフォールバックチェーンに組み込まれる

### コーディングスタイル (Coding Style)

- **言語ルール**:
  - ドキュメント・説明コメント：日本語
  - コード・変数名・関数名：英語
  - コミットメッセージ：日本語（バージョン管理プロトコルに準拠）

- **構造と配置**:
  - 新規コードは既存の機能単位（Audio / Data / UIなど）に従って配置する
  - 同一責務のコードはファイル内で近接させる

- **DRY原則**:
  - 既存関数を必ず優先利用する
  - 同一責務の処理を新規実装してはならない

- **既存コード確認ルール（重要）**:
  - コード生成前に、関連する既存処理（同一責務・類似機能）を重点的に確認すること
  - 特に以下のカテゴリは必ず確認する：
    - データ保存 / 取得処理
    - UI更新処理
    - 音声制御処理
  - 存在する場合：
    - 新規実装は禁止
    - 既存の関数を再利用または拡張して対応する

- **CSSルール（重要）**:
  - **CSS変数（`--xxx`）の使用は禁止**
  - 理由：iOS環境でのテキストコピー時にスタイル崩れ（レイアウト破綻）を引き起こすリスクがあるため

- **CSS共通化ルール（代替手段）**:
  - スタイルの再利用は以下の方法で行う：
    - 共通クラス（例：`.btn-primary`）
    - Utilityクラス（例：`.flex-center`, `.mt-8`）
  - スタイルの単純なコピペは禁止する
  - インラインスタイルの多用は禁止する（JavaScriptからの動的な値変更時のみ許可）

- **AI実行ルール（最重要）**:
  - 明示されていない機能・変更は実行しない
  - 既存コードの意図・挙動を変更しない
  - 正常に動作している処理を「リファクタリング目的」等で勝手に書き換えない
  - 指示された目的を達成するための「最小の変更」にとどめる
    
- **Git運用ルール（重要）**:
  - 通常の開発は `main` ブランチ上で行う
  - Claude Code on the web が自動生成する feature ブランチ（例: `claude/...`）は許容するが、手動でのブランチ作成やプルリクエストの提案は行わない
  - 作業完了後は、遅滞なくコミット・プッシュを実行すること

## バージョン管理プロトコル (Version Update Protocol)

### 基本原則
- プロジェクトの構造や変更履歴は、CLAUDE.mdではなく `/docs/versions/` を唯一の参照元とする
- `/docs/versions/` は「正しい状態（安定版）」のみを記録する
- バージョン作成はユーザーの明示的な指示があった場合のみ実行する

---

### 実行トリガー（厳格）
以下のような**明示的指示があった場合のみ**バージョン処理を実行する：

- 「vXXXとして記録」
- 「vXXXに上げて」
- 「この状態をvXXXとして保存」

※ 上記を含む明確なバージョン指定の文脈でのみ発動し、意図推測によるバージョン作成は禁止する

---

### 明確な禁止事項
以下はすべて禁止する：

- ユーザー指示なしで `/docs/versions/vXXX/` を作成すること
- コード修正・改善に伴い自動でバージョンを上げること
- 最新バージョンを推測して無断で更新すること
- 「良さそうだから」などのAI側の判断でバージョンを作ること
- 既存バージョンを無断で上書きすること

---

### バージョン整合ルール
- 新規作成する vXXX は、既存の最大バージョンより大きい値でなければならない
- 既存の最大バージョンより小さい、または同一の番号が指定された場合：
  → 処理を直ちに停止し、ユーザーに確認する
- バージョン番号の飛び（例：v163 → v200）は許可するが、必ずユーザー意図として扱う

---

### 状態固定ルール（追加・重要）
- バージョン生成は、現在のリポジトリのHEAD状態（コミット済みの最新状態）のみを対象とする
- 未保存・未コミットの変更が存在する場合：
  → 処理を停止し、ユーザーに確認する
- 作業途中の状態をバージョンとして記録してはならない

---

### 新バージョン (vXXX) 指定時の実行手順
ユーザーから明示的に vXXX が指定された場合のみ、以下を順次実行する：

1. `/docs/versions/` 内の最大バージョンを確認する
2. バージョン整合ルールに違反しないか検証する
3. HEAD状態がクリーンであることを確認する
4. `/docs/versions/vXXX/` を作成
5. `structure.json` を生成
   - 階層を深くせず、シンプルなファイルリストとして生成する
   - 実在するファイルのみ記載（存在しないファイルの予測追加は禁止）
   - 各ファイル/機能に以下の `status` を必ず付与する：
     - `implemented` (実装済み)
     - `partial` (一部実装)
     - `not_implemented` (未実装)
6. 直前バージョン（最大バージョン）と比較し、構造差分に矛盾がないか確認する
   - **[重要]** 意図しないファイルの消失等、不明瞭な差分がある場合は推測で補完・修正せず、必ず処理を停止してユーザーに確認する
7. `notes.md` を日本語で作成
   - 変更点 / 決定事項 / 未実装 を簡潔に記述
   - 技術的事実のみ記載（推測や曖昧な表現は禁止）
8. 作業完了後、「docs: vXXXのバージョンドキュメント生成」といった簡潔な日本語メッセージで作業ブランチへコミット・プッシュする

---

### 状態認識ルール
- `/docs/versions/` 内の最大の vXXX を常に最新とする
- `structure.json` を唯一の構造的事実として扱う
- `notes.md` は補足情報として扱う
- 不明な情報は推測で補完しない

---

### 上書き・競合制御
- 同一バージョン（vXXX）のディレクトリが既に存在する場合：
  → 新規作成・上書き・ファイル変更は一切行わず、必ずユーザーに確認する

---

### 運用制約
- 不明点がある場合は直ちに処理を停止し、ユーザーに確認する
- 通常のコード修正・軽微な変更（Gitコミット）では、本バージョン操作を行わない

---

### 補足（重要な意図）
- バージョン記録は「作業ログ」ではなく「安定状態のスナップショット」である
- 途中状態・失敗状態は記録せず、細かな履歴はGitのコミット履歴にのみ委ねる
